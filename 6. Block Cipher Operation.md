# Modes of Operation
블록 암호는 기본적으로 정해진 크기의 데이터만 처리할 수 있다. 예를 들어 DES는 64비트 단위로만 암호화를 수행한다. 하지만 실제 환경에서는 파일, 메시지, 네트워크 데이터처럼 길이가 다양한 데이터를 다뤄야 하기 때문에, 단일 블록만 처리하는 방식으로는 부족하다.

이 문제를 해결하기 위해 등장한 개념이 “mode of operation”이다. 이는 하나의 블록 암호 알고리즘을 여러 번 반복해서 적용함으로써, 블록보다 큰 데이터를 안전하게 암호화할 수 있도록 해주는 규칙 또는 방식이다. 즉, 단순히 블록을 나누는 것이 아니라, 각 블록을 어떤 순서와 방식으로 암호화하고 서로 어떻게 연결할지를 정의하는 것이다.

결국 핵심은, 블록 암호 자체는 “한 덩어리 처리 도구”이고, mode of operation은 그 도구를 사용해서 “큰 데이터를 처리하는 방법”이라고 보면 된다.

![](../images/Pasted%20image%2020260414140153.png)

기밀성을 보장하는 알고리즘이 몇 가지 있고, 기밀성와 무결성을 둘 다 보장할 수 있을 알고리즘이 있다.

# Padding
위에서 다뤘듯이 블록 암호는 정해진 크기의 데이터만 처리할 수 있기 때문에, 평문의 길이가 블록 크기와 맞지 않는 경우가 발생한다. 이때 사용하는 것이 padding이며, 쉽게 말해 부족한 공간을 채워 넣는 과정이라고 보면 된다. padding에도 여러 가지 방식이 존재한다.

## Zero Padding

![](../images/Pasted%20image%2020260414134542.png)

가장 단순한 방법은 남는 부분을 전부 0으로 채우는 것이다. 구현이 매우 간단하다는 장점이 있지만, 문제가 있다. 만약 원래 평문의 마지막 값이 0으로 끝난다면, 어디까지가 실제 데이터이고 어디서부터 padding이 시작된 것인지 구분할 수 없다. 즉, 복호화 시 원본을 정확하게 복원하기 어려운 상황이 발생할 수 있다.
## Bit Padding

이러한 문제를 해결하기 위해 나온 방법이 bit padding이다. 

![](../images/Pasted%20image%2020260414134700.png)

기본적으로 0으로 채우는 것은 동일하지만, padding의 시작을 명확하게 구분하기 위해 첫 비트를 1로 설정한다. 즉, `1000 0000 (0x80)`을 시작으로 그 뒤를 0으로 채운다. 이렇게 하면 복호화 시 뒤에서부터 처음 등장하는 1을 기준으로 padding의 시작 위치를 판단할 수 있다.

## Byte Padding ( ANSI X.923 )

하지만 bit padding 역시 완벽한 방법은 아니다. 특정 경우에는 여전히 모호성이 발생할 수 있다. 이를 보완한 방식이 ANSI X.923이다. 
![](../images/Pasted%20image%2020260414135129.png)

이 방식은 남는 공간을 모두 0으로 채운 뒤, 마지막 바이트에 padding의 길이를 기록한다. 예를 들어 3바이트를 padding했다면 마지막 값은 `03`이 된다. 따라서 복호화 시 마지막 바이트만 확인하면 몇 바이트를 제거해야 하는지 명확하게 알 수 있다.

## Byte Padding ( PKCS #7 )
PKCS #7 방식은 ANSI X.923과 유사하지만, padding을 채우는 방식이 다르다. 단순히 0으로 채우는 것이 아니라, padding한 바이트 수 자체를 모든 padding 위치에 동일하게 채운다. 

![](../images/Pasted%20image%2020260414135403.png)

예를 들어 3바이트를 padding해야 한다면 `03 03 03`으로 채운다. 이 방식은 마지막 바이트뿐만 아니라 전체 padding 영역이 동일한 값을 가지기 때문에, 검증과 복원이 더 직관적이고 안정적으로 이루어진다.

# No-Padding : CTS
패딩을 사용하는 이유는 평문의 길이를 블록 크기에 맞추기 위해서다. 하지만 반드시 패딩을 사용해야 하는 것은 아니고, 패딩 없이 처리하는 방법도 존재하는데 그 중 하나가 Ciphertext Stealing(CTS) 방식이다.

일반적으로 평문을 블록 단위로 나누면, 앞부분은 블록 크기와 정확히 맞아서 정상적으로 암호화가 가능하지만, 마지막 블록은 길이가 부족해서 그대로는 암호화할 수 없는 경우가 발생한다. 기존 방식에서는 이 부족한 부분을 padding으로 채웠지만, CTS는 이를 다른 방식으로 해결한다.

![445](../images/Pasted%20image%2020260414135527.png)

CTS에서는 먼저 마지막 이전 블록까지는 정상적으로 암호화를 수행한다. 그리고 마지막에 남은 부족한 평문 블록에 대해서는, 바로 앞에서 암호화된 블록의 일부를 가져와 길이를 맞춘 뒤 암호화를 진행한다. 이후, 이 과정에서 사용된 데이터의 일부를 다시 재배치하여 앞뒤 블록의 위치를 조정한다.

결과적으로, 추가적인 padding 없이도 전체 암호문 길이를 원래 평문 길이와 동일하게 유지할 수 있다. 즉, CTS의 핵심은 부족한 블록을 채우기 위해 별도의 padding을 추가하는 대신, 이미 생성된 암호문 일부를 “빌려와서(stealing)” 사용하는 데 있다.

# Confidential Only

이제 기밀성(confidentiality)을 보장하기 위해 사용되는 몇 가지 암호화 모드에 대해 살펴보자.

## Electronic Codebook(ECB)
ECB 방식은 평문을 여러 개의 블록으로 나눈 뒤, 각 블록을 동일한 키를 사용하여 서로 독립적으로 암호화하는 방식이다. 즉, 하나의 블록을 암호화할 때 다른 블록의 정보는 전혀 사용되지 않는다.

![](../images/Pasted%20image%2020260414140338.png)

그림에서 볼 수 있듯이, 모든 평문 블록은 같은 키로 각각 암호화되며, 복호화 역시 동일한 키를 사용하여 각 블록을 독립적으로 수행한다. 구조 자체는 단순하고 이해하기 쉽다.

![](../images/Pasted%20image%2020260414140446.png)

하지만 이 방식은 직관적으로도 문제가 존재한다. 동일한 평문 블록이 입력되면 항상 동일한 암호문 블록이 출력된다. 이는 모든 블록이 같은 키로, 그리고 서로 영향을 주지 않고 독립적으로 처리되기 때문이다. 따라서 평문에 반복되는 패턴이 존재할 경우, 암호문에서도 동일한 패턴이 그대로 나타난다.

이러한 특성 때문에 ECB는 데이터의 구조나 패턴을 숨기지 못하며, 보안적으로 취약한 방식으로 평가된다. 실제로 이미지와 같은 데이터를 ECB로 암호화하면, 암호화 이후에도 원래 형태가 어느 정도 드러나는 현상이 발생한다.
