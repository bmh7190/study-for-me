Bottom-up parsing 방식의 대표적인 기법으로 **LR(k) 파싱**이 있으며, 일반적으로 특별한 언급이 없으면 **lookahead가 1인 LR(1)** 구조가 많이 사용된다. 

LR 파서의 장점으로는 **대부분의 Context-Free Grammar(CFG)로 표현되는 프로그래밍 언어를 처리할 수 있다는 점**, **백트래킹이 필요 없다는 점**, 그리고 **구문 오류(syntactic error)를 가능한 한 빠르게 탐지할 수 있다는 점**이 있다. 또한 **LL 파서로 파싱 가능한 모든 언어를 LR 파서로도 처리할 수 있다.** 

다만, **파서를 설계하고 파싱 테이블을 생성하는 과정이 복잡하여 많은 노력이 필요하다는 단점**이 있다.

# User push down automata
LR Parser는 Pushdown Automata(PDA)를 기반으로 동작하며, 스택을 이용하여 문맥 자유 문법(Context-Free Grammar, CFG)을 효율적으로 파싱한다. 

파서는 다양한 상태(state)를 정의하고, 이러한 상태들을 바탕으로 parsing table을 구성한다. 즉, 현재 상태와 다음 입력 심볼에 따라 어떤 동작을 수행할지 결정하게 된다. 

이때 parsing table은 Action table과 Goto table로 구성되는데, Action table은 현재 상태와 다음 입력 심볼이 주어졌을 때 shift, reduce, accept, error 중 어떤 동작을 수행할지를 정의하며, Goto table은 reduction 이후 특정 비단말 기호에 대해 어떤 상태로 전이해야 하는지를 나타낸다. 

이러한 구조를 통해 LR 파서는 하나의 상태 기계로 동작한다. 비록 CFG 자체는 Finite Automata로 직접 변환될 수 없지만, LR 파싱에서는 handle을 포함하는 viable prefix를 인식하는 DFA를 구성할 수 있다.

# Model of an LR Parser

![](../images/Pasted%20image%2020260417000851.png)

LR Parser가 동작하는 전체 모델은 다음과 같이 설명할 수 있다. 입력 스트림(input stream)에서 토큰을 하나씩 읽어 LR Parsing Program(Driver)으로 전달하며, 이 과정에서 스택을 이용하여 파싱이 진행된다. 파서는 현재 스택의 상태와 다음 입력 심볼을 기반으로 parsing table을 참조하여 어떤 동작을 수행할지 결정한다. 이때 수행 가능한 동작은 **shift**, **reduce**, **accept**, **error**의 네 가지이다. Shift는 다음 입력 심볼을 스택에 추가하는 동작이며, Reduce는 스택의 상단에 있는 심볼들이 특정 생산 규칙의 오른쪽 부분과 일치할 때 이를 왼쪽 부분의 비단말 기호로 치환하는 동작이다. Accept는 입력이 문법적으로 올바를 때 파싱을 종료하는 것을 의미하며, Error는 문법 오류가 발생했음을 나타낸다.

또한, LR 파서는 **Action table**과 **Goto table**을 사용하여 상태 전이를 결정한다. Action table은 현재 상태와 다음 입력 심볼에 따라 수행할 동작을 정의하며, Goto table은 Reduce 이후 비단말 기호에 대해 어떤 상태로 이동할지를 나타낸다. 이러한 상태 전이는 LR(0), SLR, LR(1), LALR(1) 방법을 통해 구성된 DFA(Deterministic Finite Automaton)를 기반으로 이루어진다.

스택에는 단순히 문법 기호만 저장되는 것이 아니라, 각 기호가 스택에 추가될 때의 **상태(state)** 도 함께 저장된다. 실제 구현에서는 상태 번호만 저장해도 되지만, 설명의 편의를 위해 문법 기호와 상태를 함께 표현하기도 한다. 이러한 구조를 통해 LR Parser는 현재 상태와 입력 심볼만으로 다음 동작을 결정할 수 있으며, 백트래킹 없이 효율적인 구문 분석을 수행할 수 있다.

# LR Parsing Driver
그림에서 나타난 **driver**는 LR 파서의 핵심 구성 요소로, 파싱 과정을 실제로 수행하는 제어 프로그램을 의미한다.

![](../images/Pasted%20image%2020260417001216.png)

Driver는 입력 스트림, 스택, 그리고 parsing table(Action과 Goto)을 이용하여 파싱을 진행한다. LR 파서의 현재 상태는 스택의 내용과 남아 있는 입력으로 표현되며, 일반적으로 `(s₀ X₁ s₁ X₂ s₂ … Xₘ sₘ, aᵢ aᵢ₊₁ … aₙ $)`와 같은 형태의 configuration으로 나타낸다. 여기서 스택에는 문법 기호와 상태가 번갈아 저장되며, 실제 구현에서는 상태 번호만 저장해도 되지만 설명의 편의를 위해 두 정보를 함께 표현한다. 입력은 아직 처리되지 않은 토큰들의 나열과 끝을 의미하는 `$`로 구성된다.

Driver는 스택의 최상단 상태 `sₘ`과 다음 입력 심볼 `aᵢ`를 이용하여 `action[sₘ, aᵢ]` 값을 참조하고, 그 결과에 따라 동작을 결정한다. 만약 `action[sₘ, aᵢ] = shift s`라면, 입력 심볼 `aᵢ`와 새로운 상태 `s`를 스택에 push하고 입력 포인터를 다음 심볼로 이동시킨다. `action[sₘ, aᵢ] = reduce A → β`인 경우에는 `β`의 길이를 `r = |β|`라 할 때 스택에서 `2r`개의 항목을 pop한 후, 비단말 기호 `A`를 push하고 `goto[sₘ₋ᵣ, A]`를 참조하여 얻은 새로운 상태 `s`를 다시 스택에 push한다. `action[sₘ, aᵢ] = accept`이면 입력 문자열이 문법적으로 올바르다는 의미이므로 파싱을 종료하고, `action[sₘ, aᵢ] = error`인 경우에는 구문 오류가 발생했음을 의미하며 오류 복구 절차를 시도하게 된다.

결과적으로 driver는 현재 상태와 다음 입력 심볼을 기반으로 shift, reduce, accept, error의 네 가지 동작을 결정하며, Action과 Goto 테이블을 이용해 상태 전이를 수행하는 LR 파서의 제어 장치라고 할 수 있다. 이를 통해 LR 파서는 백트래킹 없이 결정적인 방식으로 효율적인 구문 분석을 수행하게 된다.

# Example LR(0) parsing table
사실 이렇게 보면 이해가 안될 수 있기 때문에 예시를 통해 알아보자

![](../images/Pasted%20image%2020260417001548.png)

Parsing table을 DFA와 그래머를 기반으로 구성한다고 생각해보자. 먼저 **state 1**을 보면, 입력 기호가 `$`일 경우 파싱이 완료되었음을 의미하므로 action 테이블의 `$` 열에는 `accept(acc)`가 들어간다. 또한 입력 기호가 `a`일 경우 DFA에서 다음 상태로 전이되므로, action 테이블의 `a` 열에는 `s2`(shift 후 state 2로 이동)가 기록된다.

이제 **state 2**를 살펴보자. 여기서 goto 테이블의 `A` 열에 `g3`가 있다는 것은 `goto[2, A] = 3`을 의미하며, **비단말 기호 A가 스택에 추가된 이후 state 3으로 전이한다는 뜻**이다. 이는 **reduce 동작 자체를 의미하는 것이 아니라**, reduce가 수행된 이후에 적용되는 상태 전이를 나타낸다. 즉, 어떤 상태에서 `A → β`와 같은 생산 규칙으로 reduce가 일어나고 나면, 스택의 새로운 top 상태를 기준으로 `goto` 테이블을 참조하여 다음 상태로 이동하게 되며, 그 결과가 `g3`이다.

그리고 하나 더 살펴보면, **state 4**에서는 action 테이블의 `a, b, c, d, e, $` 모든 입력 기호에 대해 `r3`가 들어가 있는 것을 확인할 수 있다. 이는 **state 4에 도달하면 다음 입력 심볼이 무엇이든지 간에 그래머의 3번 생산 규칙에 따라 reduce를 수행해야 함을 의미한다.**

여기서 `r3`이라는 것은 **그래머의 3번 생산 규칙에 따라 reduce가 수행된다는 의미**이다. 예를 들어, 3번 규칙이 `A → b`라면, 스택의 top에 있는 `b`를 pop하고 이를 비단말 기호 `A`로 치환하게 된다.

이후 `A`가 스택에 추가되면, 다음 상태로의 전이는 **goto table**을 참조하여 결정된다. 즉, reduce가 수행된 후 스택의 새로운 top 상태를 기준으로 `goto[현재 상태, A]`를 확인하게 되며, 그 결과가 `g3`이라면 이는 **state 3으로 이동하라는 의미**이다. 따라서 스택에는 비단말 `A`와 함께 상태 `3`이 push된다.

정리하면, **state 4에서는 항상 생산 규칙 3번으로 reduce가 수행되며**, 그 이후의 상태 전이는 goto 테이블의 결과에 따라 결정된다. 이 예시에서는 `goto[2, A] = 3`이므로, reduce 이후 state 3으로 이동하게 된다.


입력에 따른 스택의 변화를 살펴보자. 스택에는 처음에 초기 상태인 **1**만 들어가 있다. 

![](../images/Pasted%20image%2020260417002638.png)

이후 입력 심볼 `a`를 읽으면 action 테이블에 따라 `s2`가 수행되므로, `a`와 함께 상태 **2**가 스택에 추가된다. 다음 입력인 `b`가 들어오면 역시 `s4`가 적용되어 `b`와 상태 **4**가 스택에 추가된다.

이때 **state 4**에서는 `r3`가 적용되는데, 이는 그래머의 3번 생산 규칙(예: `A → b`)에 따라 **reduce**가 수행됨을 의미한다. Reduce 과정에서는 먼저 스택에서 상태 **4**와 심볼 `b`를 pop한 후, 해당 규칙의 왼쪽 부분인 비단말 기호 `A`를 스택에 push한다. 그 다음, 스택의 새로운 top 상태를 기준으로 **goto table**을 참조하게 되며, `goto[2, A] = 3`이므로 상태 **3**이 스택에 추가된다.

따라서 이 과정은 초기 상태 **1**에서 시작하여 `a` 입력으로 인해 **2**로 이동하고, `b` 입력으로 **4**까지 확장된 뒤, `r3`에 의한 reduce와 `g3`에 따른 상태 전이를 통해 최종적으로 **3**으로 이동하는 구조라고 할 수 있다.

# Issue in the Shift - Reduce Parser
다음 parser로 넘어가기 전에, 앞에서 다룬 **단순 shift-reduce parser**의 경우 몇 가지 문제점이 있었다. 가장 큰 문제는 **shift와 reduce 중 어떤 동작을 선택해야 하는지에 대한 명확한 기준이 없다는 점**이다. 어떤 시점에서는 당장 reduce를 수행할 수 있지만, 최종적으로 올바른 파싱 결과를 얻기 위해서는 **reduce를 미루고 shift를 선택해야 하는 경우**도 존재한다. 이러한 상황에서는 잘못된 선택을 할 가능성이 있으며, 그 결과를 수정하기 위해 **백트래킹(backtracking)** 이 필요할 수 있다.

![](../images/Pasted%20image%2020260417003018.png)

그래서 이러한 문제를 해결하기 위해, **LR(0) automata**를 이용해 현재 파서의 상태를 저장하고, 여기에 **FOLLOW 집합**을 함께 활용하여 다음 동작을 결정하는 방식으로 문제를 해결할 수 있다. 즉, 단순히 스택의 모양만 보고 shift할지 reduce할지를 판단하는 것이 아니라, **현재 어떤 상태에 있는지**와 **앞으로 어떤 입력이 올 수 있는지**를 함께 고려하여 보다 체계적으로 동작을 결정하는 것이다. 이러한 과정을 통해 shift-reduce parser에서 발생하던 모호성과 비효율성을 줄일 수 있다. 이제 이것이 구체적으로 어떤 의미인지 차근차근 살펴보자.

---
# SLR Grammer

**SLR(Simple LR)** 은 _Simple LR_의 약자로, **LR(0) shift-reduce parsing의 간단한 확장 버전**이다. LR(0) 파서는 상태 정보만을 이용하여 shift와 reduce를 결정하기 때문에, 특정 상황에서 **shift/reduce conflict**나 **reduce/reduce conflict** 가 발생할 수 있다.

SLR 파서는 이러한 충돌을 완화하기 위해 **FOLLOW 집합**을 활용한다는 점에서 LR(0)과 차이가 있다. 즉, 어떤 상태에서 `A → α`와 같은 **reduce 항목**이 존재할 때, LR(0)에서는 모든 입력 기호에 대해 reduce를 수행할 수 있지만, SLR에서는 **FOLLOW(A)** 에 속하는 입력 기호에 대해서만 reduce를 수행하도록 제한한다.

![](../images/Pasted%20image%2020260417003553.png)

# SLR Parsing Table

![](../images/Pasted%20image%2020260417004056.png)

왼쪽의 **LR(0) parsing table**을 보면, **state 2**에서 입력 심볼 `+`에 대해 `s3/r3`로 표시되어 있다. 이는 **shift/reduce conflict**가 발생했음을 의미한다. 즉, 파서는 `+`를 읽고 **state 3으로 shift**해야 할지, 아니면 생산 규칙 3번인 `E → id`로 **reduce**해야 할지를 결정할 수 없는 모호한 상황에 놓이게 된다. 이러한 충돌은 LR(0) 파서가 **lookahead 정보 없이** 상태 정보만을 기반으로 동작하기 때문에 발생한다.

반면, 오른쪽의 **SLR parsing table**에서는 이러한 충돌이 해결된 것을 확인할 수 있다. SLR 파서는 **FOLLOW 집합**을 활용하여 reduce 동작을 제한한다. 이 예제에서 `FOLLOW(E) = { $ }`이므로, 생산 규칙 `E → id`에 대한 reduce(`r3`)는 다음 입력 심볼이 `$`일 때에만 수행된다. 따라서 **state 2에서 입력이 `+`인 경우에는 reduce가 아닌 shift(`s3`)만 수행**하게 되어 충돌이 제거된다.

정리하면, LR(0) 테이블에서는 lookahead 정보가 없기 때문에 `+`에 대해 `s3/r3`와 같은 충돌이 발생하지만, SLR 테이블에서는 FOLLOW 집합을 이용하여 reduce를 `$`에서만 허용함으로써 `+`에서는 shift, `$`에서는 reduce가 이루어지도록 하여 충돌을 해결한다.

# SLR Parsing 
SLR 파서를 이해하기 위해서는 **LR(0) Item**이라는 새로운 개념이 등장한다. **LR(0) Item**이란 Production Rule의 오른쪽 부분에 점(dot, •)이 포함된 형태를 의미한다. 이 점은 해당 생산 규칙에서 현재 파싱이 어느 위치까지 진행되었는지를 나타내며, 파서가 문법 규칙 내에서 어떤 단계에 있는지를 표현한다. 즉, LR(0) Item은 파싱 과정에서 **생산 규칙이 얼마나 인식되었는지**를 나타내는 일종의 상태 정보라고 할 수 있다.

예를 들어, 생산 규칙이 `E → id + E`라면, 이에 대한 LR(0) Item은 다음과 같이 표현될 수 있다.

- `E → • id + E` : 아직 아무 것도 인식되지 않은 상태
- `E → id • + E` : `id`까지 인식한 상태
- `E → id + • E` : `id +`까지 인식한 상태
- `E → id + E •` : 전체 규칙이 인식되어 reduce가 가능한 상태

이처럼 점(dot, •)의 위치는 **파싱이 진행되는 단계**를 나타내며, LR 파서의 각 상태는 이러한 LR(0) Item들의 집합으로 구성된다. 따라서 LR(0) Item은 SLR 파싱 테이블을 구성하는 LR(0) DFA의 상태를 정의하는 핵심적인 요소이다.

# Constructing SLR Parsing Tables
SLR 파싱 테이블을 구성하기 위해서는 먼저 문법을 **확장(augment)** 해야 한다. 이는 새로운 시작 기호 $S'$를 추가하여 $S' \rightarrow S$와 같은 생산 규칙을 만드는 과정으로, 파싱의 종료 조건을 명확히 하기 위함이다. 그 다음, **Closure**와 **Goto 연산**을 이용하여 LR(0) 아이템들의 집합 $C = \{ I_0, I_1, \dots, I_n \}$을 생성하고, 이를 통해 **LR(0) DFA**를 구성한다. 이때 초기 상태 $I_0$는 아이템 $[S' \rightarrow \bullet S]$를 포함하는 집합이 된다.

이후 생성된 각 상태에 대해 **Action**과 **Goto 테이블**을 채워 나간다. 어떤 상태 $I_i$에 아이템 $[A \rightarrow \alpha \bullet a \beta]$가 존재하고, `goto(I_i, a) = I_j`일 경우, 입력 기호 `a`에 대해 **shift j**를 Action 테이블에 기록한다. 또한, 상태 IiI_iIi에 아이템 $[A \rightarrow \alpha \bullet]$가 포함되어 있다면, 비단말 $A$의 **FOLLOW(A)** 에 속하는 모든 터미널 $a$에 대해 **reduce A→αA** 를 Action 테이블에 기록한다. 단, $A = S'$인 경우는 제외한다 

만약 상태 $I_i$에 아이템 $[S' \rightarrow S \bullet]$가 포함되어 있다면, 입력 기호 `$`에 대해 **accept**를 Action 테이블에 기록한다. 그리고 `goto(I_i, A) = I_j`인 경우에는 비단말 $A$에 대해 **goto[i, A] = j**를 Goto 테이블에 기록한다. 이러한 과정을 더 이상 추가할 항목이 없을 때까지 반복하면 SLR 파싱 테이블이 완성된다.
