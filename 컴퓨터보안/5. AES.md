DES는 56비트의 짧은 키 길이로 인해 브루트 포스 공격에 취약해졌고, 이를 보완하기 위해 Double DES와 Triple DES가 제안되었다. 그러나 Triple DES 역시 64비트 블록 크기의 한계와 낮은 성능으로 인해 점차 사용이 감소하게 되었다. 이러한 문제를 해결하기 위해 새로운 표준 암호 알고리즘의 필요성이 제기되었고, 그 결과 **AES(Advanced Encryption Standard)** 가 등장하게 되었다.

# AES Requirements

AES(Advanced Encryption Standard)는 DES와 Triple DES의 한계를 극복하기 위해 NIST가 제정한 대칭키 블록 암호 표준이다. AES 알고리즘이 선정되기 위해서는 다음과 같은 요구사항을 만족해야 했다.

#### 1. Symmetric Block Cipher
AES는 **대칭키 블록 암호**로 설계되었으며, 다음과 같은 특징을 가진다.

- **블록 크기**: 128비트
- **키 길이**: 128비트, 192비트, 256비트
- 동일한 키를 사용하여 암호화와 복호화를 수행한다.

#### 2. Stronger and Faster than Triple-DES
AES는 기존의 Triple DES보다 **보안성과 성능 측면에서 우수**해야 했다.

- 더 긴 키 길이를 지원하여 높은 보안성 제공
- 128비트 블록 크기를 사용하여 보안성 향상
- 소프트웨어와 하드웨어 환경에서 **빠른 처리 속도**를 제공

#### 3. Provide Full Specification and Design Details
AES는 **완전한 명세와 설계 세부 사항**을 제공해야 했다.

- 알고리즘의 구조가 공개되어야 하며
- 누구나 구현하고 검증할 수 있도록 투명성이 보장되어야 했다.
- 이는 보안 알고리즘의 신뢰성을 높이는 중요한 요소이다.

#### 4. Both C and Java Implementations
AES는 다양한 환경에서 사용될 수 있도록 **C와 Java에서의 구현 가능성**이 요구되었다.

- 소프트웨어 구현의 효율성
- 다양한 플랫폼에서의 호환성 확보

#### 5. Shortlisted Algorithms (Aug-1999)
NIST는 1997년 공모를 통해 여러 후보 알고리즘을 평가하였고, 1999년 8월에 다음의 다섯 개 알고리즘을 최종 후보로 선정하였다.

| 알고리즘                   | 특징                         |
| ---------------------- | -------------------------- |
| **MARS (IBM)**         | 복잡하지만 빠르며 높은 보안 여유도        |
| **RC6 (USA)**          | 단순하고 빠르지만 보안 여유도는 상대적으로 낮음 |
| **Rijndael (Belgium)** | 구조가 간결하고 빠르며 적절한 보안 여유도    |
| **Serpent (Europe)**   | 느리지만 매우 높은 보안 여유도          |
| **Twofish (USA)**      | 복잡하지만 빠르고 높은 보안 여유도        |

이 중 **Rijndael** 알고리즘이 최종적으로 선택되어 **2001년 FIPS PUB 197**로 표준화되었으며, 현재 우리가 사용하는 AES가 되었다.

# AES Overview
AES(Advanced Encryption Standard)는 **대칭키 블록 암호**로, 고정된 크기의 평문 블록을 입력받아 동일한 크기의 암호문 블록을 출력한다. AES의 외부적인 구조는 입력 데이터와 키의 크기, 그리고 라운드 수와 같은 주요 파라미터로 설명할 수 있다.

# **AddRoundKey**

- 상태(state)를 128비트 라운드 키와 **XOR 연산**하여 결합한다.
- 연산은 **열(column) 단위**로 수행되며, 각 state의 열에 대응되는 라운드 키 워드가 더해진다.
- 이 과정은 **행렬 덧셈 형태**로 볼 수 있다.
- 복호화에서도 동일한 연산이 사용된다.  XOR 연산은 자기 자신이 역연산이므로, **라운드 키 순서만 반대로 적용하면 되기 때문이다.**
- 전체 AES 과정 중 가장 단순한 단계로 설계되었다.

# Key Scheduling

**AES Key Scheduling**은 128비트(16바이트)의 초기 키를 입력으로 받아, 이를 여러 개의 32비트 워드(총 44/52/60개)로 확장하는 과정이다. 

먼저 초기 키를 그대로 복사하여 첫 4개의 워드(w₀ ~ w₃)를 구성한 뒤, 이후에는 반복적으로 새로운 워드를 생성한다. 각 워드는 기본적으로 바로 이전 워드(wᵢ₋₁)와 4칸 전 워드(wᵢ₋₄)에 의존하여 만들어지며, 대부분의 경우 단순히 두 값을 XOR하여 계산된다. 그러나 4의 배수 위치에 해당하는 워드를 생성할 때는 추가적인 변환이 적용된다. 

이 경우, 먼저 이전 워드에 대해 바이트 단위 순환 이동(RotWord)을 수행하고, 이어서 S-box를 이용한 치환(SubWord)을 적용한 뒤, 라운드 상수(Rcon)와 XOR 연산을 한다. 마지막으로 이 결과를 wᵢ₋₄와 다시 XOR하여 새로운 워드를 생성한다. 

이러한 과정을 통해 단순한 XOR 구조에 주기적으로 비선형성과 라운드 의존성을 추가함으로써, 최종적으로 생성되는 라운드 키들이 보다 복잡하고 안전한 형태를 갖도록 한다.

## AES Key Scheduling

![](../images/Pasted%20image%2020260419160008.png)
이 그림은 AES에서 사용되는 초기 키를 **워드(word) 단위로 재구성하고, 이후 키 스케줄링을 통해 확장하는 과정의 시작**을 나타낸다. 왼쪽에 보이는 4×4 배열은 128비트 키를 바이트 단위로 표현한 것으로, 각 원소는 1바이트를 의미한다. AES에서는 이 키를 그대로 사용하는 것이 아니라, **열(column) 단위로 묶어서 32비트 워드로 변환**한다. 즉, 첫 번째 열은 ($w_0$), 두 번째 열은 ($w_1$), 세 번째 열은 ($w_2$), 네 번째 열은 ($w_3$)가 되며, 각각 4바이트로 이루어진 하나의 워드를 구성한다.

이렇게 생성된 초기 4개의 워드는 키 스케줄링의 입력으로 사용되며, 이후 반복적인 연산을 통해 ($w_4, w_5, \dots$)와 같은 새로운 워드들이 순차적으로 생성된다. 그림의 오른쪽에 표시된 ($w_{42}, w_{43}$)는 이러한 과정을 통해 확장된 최종 단계의 워드를 의미한다. 

## AES Key Expansion

![](../images/Pasted%20image%2020260419160213.png)

AES에서는 하나의 ($k_i$)가 1바이트를 의미하므로, 전체 키는 총 16바이트, 즉 128비트로 구성된다. 초기 단계에서는 이 128비트 키를 그대로 사용하여 4개의 워드($w_0, w_1, w_2, w_3$)를 생성하며, 각 워드는 4바이트로 이루어진다.

이후 키 스케줄링 과정에서는 새로운 워드를 생성하기 위해 이전 워드들을 기반으로 연산을 수행한다. 기본적으로는 바로 이전 워드와 4칸 앞의 워드를 XOR하여 새로운 워드를 만들어 나간다. 

다만, 특정 위치(4의 배수)에 해당하는 워드를 생성할 때는 단순 XOR이 아니라, 먼저 임시 값 ($t_i$)를 생성한 후 이를 사용한다. 이 ($t_i$)는 이전 워드에 대해 RotWord(순환 이동), SubWord(S-box 치환), 그리고 라운드 상수(Rcon)와의 XOR 연산을 거쳐 만들어진다.


1. 먼저 **RotWord** 연산을 통해 워드 내 바이트들을 한 바이트씩 왼쪽으로 순환 이동시킨다. 
	예를 들어, $[b_0, b_1, b_2, b_3]$는 $[b_1, b_2, b_3, b_0]$로 변환된다. 

2. 이후 **SubWord** 연산을 적용하여, 각 바이트를 S-box를 이용해 치환함으로써 비선형성을 추가한다. 

3. 마지막으로 이 결과값에 **라운드 상수(Rcon)** 를 XOR 연산으로 결합하여 임시 값이 생성된다. 이러한 과정을 통해 단순한 선형 구조에 변화를 주고, 키 확장 과정에 보안성을 강화한다.

## Round Constant (RCon)

AES 키 스케줄링에서 사용되는 **Rcon(Round Constant)**은 하나의 워드 형태로 표현되며, 오른쪽 3바이트는 모두 0으로 채워지고, 왼쪽 1바이트만 의미 있는 값을 가진다. 

즉, $Rcon[j] = (r_j, 0, 0, 0)$ 형태로 구성된다.

이 값은 라운드마다 서로 다르게 정의되며, 초기값은 $Rcon[1] = 1$이고 이후 값들은 이전 값에 2를 곱하는 방식으로 생성된다. 이때의 곱셈은 일반적인 정수 연산이 아니라 **GF($2^8$) 유한체 상에서의 연산**으로 이루어진다. 따라서 단순한 비트 연산이 아니라, 필요할 경우 기약 다항식을 기준으로 한 모듈러 연산이 포함된다.

실제 구현에서는 이러한 연산을 직접 계산하기보다는, 미리 정의된 값을 **테이블 형태로 저장하여 lookup 방식으로 사용하는 경우가 일반적**이다.

# AES Decryption
AES 복호화는 암호화 과정을 단순히 그대로 따라가는 것이 아니라, **각 단계의 역연산을 역순으로 적용하는 방식**으로 수행된다. 따라서 암호화와 동일하지는 않지만, 각 연산이 모두 가역적(invertible)이기 때문에 이를 이용해 복호화가 가능하다.

또한 구현 측면에서는, 복호화를 암호화와 유사한 구조로 처리하기 위해 **각 단계의 역연산을 사용한 ‘equivalent inverse cipher’** 를 정의할 수 있다. 이 경우 전체적인 흐름은 암호화와 거의 동일하지만, SubBytes 대신 InvSubBytes, ShiftRows 대신 InvShiftRows, MixColumns 대신 InvMixColumns와 같은 역연산이 사용되며, 라운드 키 역시 다른 순서로 적용된다.

흥미로운 점은 일부 연산의 순서를 바꾸어도 결과가 동일하게 유지된다는 것이다. 

예를 들어, SubBytes과 ShiftRows, 그리고 MixColumns과 AddRoundKey는 특정 조건에서 서로 순서를 교환해도 동일한 효과를 낼 수 있다. 이러한 성질 덕분에 AES 복호화는 구조적으로 암호화와 유사한 형태로 구성할 수 있으며, 효율적인 구현이 가능하다.

![](../images/Pasted%20image%2020260419161645.png)

왼쪽 그림은 AES 복호화를 이론적으로 가장 직관적인 방식으로 표현한 구조로, 암호화 과정을 그대로 거꾸로 따라가며 각 단계의 역연산을 순서대로 적용하는 형태이다. 즉, 암호화에서 수행했던 연산들을 역순으로 적용해야 하므로 전체 흐름이 암호화 과정과는 다소 다르게 구성된다. 이 방식은 개념적으로 이해하기는 쉽지만, 실제 구현에서는 암호화와 별도의 구조를 가져야 하므로 코드나 하드웨어를 따로 설계해야 하는 비효율이 발생한다.

반면 오른쪽 그림은 이러한 비효율을 개선하기 위해 복호화 과정을 암호화와 유사한 구조로 재구성한 형태이다. 이 구조에서는 각 단계의 역연산을 사용하되, 라운드 키를 미리 변형하여 연산 순서를 조정함으로써 전체 흐름을 암호화와 비슷하게 맞춘다. 특히 AddRoundKey와 MixColumns의 순서를 직접 바꾸는 대신, 라운드 키에 미리 inverse mix를 적용하여 결과적으로 동일한 효과를 얻도록 만든 것이 핵심이다. 그 결과, 연산 흐름이 일정한 패턴을 가지게 되어 구현이 훨씬 단순해지고, 기존 암호화 구조를 재사용할 수 있는 장점이 생긴다.

따라서 오른쪽 구조의 가장 큰 장점은 복호화를 별도의 복잡한 과정으로 처리하는 것이 아니라, 암호화와 유사한 형태로 통일하여 **구현 효율성과 구조적 일관성을 높일 수 있다는 점**에 있다.
