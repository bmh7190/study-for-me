여기서 핵심은 “웹에서 클라이언트와 서버가 안전하게 통신하려면 어떤 구조가 필요한가?”이다.

우리가 웹사이트에 접속할 때 보통 HTTP가 아니라 HTTPS를 사용한다.  
HTTPS는 HTTP 자체가 암호화된 것이 아니라, **HTTP 데이터를 TLS 위에 올려서 전송하는 구조**라고 보면 된다.

즉, TLS는 애플리케이션 계층의 데이터를 TCP 위에서 안전하게 전달할 수 있도록 해주는 보안 계층이다.

---
# Web Security Cosiderations
웹은 기본적으로 클라이언트 / 서버 구조이다. 사용자는 크롬 같은 브라우저 같은 클라이언트를 통해 서버에 요청을 보내고, 서버는 응답을 준다.

그런데 웹 사용자는 보안 전문가가 아닌 경우가 많다. 대부분의 사용자는 네트워크 중간에서 누가 데이터를 훔쳐볼 수 있는지, 서버가 진짜 서버인지, 데이터가 변조될 수 있는지 잘 모른다.

그래서 웹 보안에서는 사용자가 직접 보안을 챙기기보다는, 프로토콜 차원에서 자동으로 보안을 제공해야 한다.

여기서 TLS가 필요한 이유가 나온다. TLS는 사용자가 복잡한 보안 절차를 직접 알지 못해도, 브라우저와 서버 사이에서 자동으로 다음을 제공한다.

- 서버 인증
- 데이터 기밀성
- 데이터 무결성
- 세션 키 생성

즉, 웹 환경에서는 보안 지식이 없는 일반 사용자도 안전하게 서비스를 사용할 수 있어야 하므로 TLS 같은 표준 보안 프로토콜이 필요하다.

---
# Relative Location of Security Facilities in TCP/IP

보안 기능은 여러 계층에 넣을 수 있다.

![](../images/Pasted%20image%2020260519180124.png)

예를 들어 네트워크 계층에서는 IPsec 같은 방식으로 보안을 제공할 수 있고, 애플리케이션 계층에서는 애플리케이션 자체가 암호화를 직접 처리할 수도 있다.

TLS는 그중에서 **전송 계층 근처**, 정확히는 애플리케이션과 TCP 사이에 위치한다고 보면 된다.
그래서 HTTPS 는 HTTP + TLS 로 HTTP 메세지를 TLS가 감싸고, TLS가 암호화한 데이터를 TCP로 전송하는 구조라고 보면된다.

# SSL / TLS Protocol
SSL/TLS는 TCP 위에서 동작하는 보안 프로토콜이다. TCP는 신뢰성 있는 전송을 제공하지만, 보안은 제공하지 않는다.

TCP가 해주는 것은 다음과 같다.

```
데이터를 순서대로, 손실 없이 전달
```

하지만 TCP만으로는 다음 문제를 해결할 수 없다.

```
누가 훔쳐보는가?
중간에서 변조되지는 않았는가?
내가 접속한 서버가 진짜 서버인가?
```

TLS는 이 문제를 해결한다. TLS가 제공하는 핵심 기능은 세 가지이다.

##### **Authentication**  
클라이언트는 서버 인증서를 확인해서 이 서버가 진짜 서버인지 검증한다.
##### **Confidentiality**  
클라이언트와 서버가 공유한 대칭키로 데이터를 암호화한다.
##### **Integrity
전송 중 데이터가 변조되었는지 확인할 수 있다.

정리하면 TLS는 TCP 통신 위에 보안 기능을 얹어서, HTTP 같은 애플리케이션 데이터를 안전하게 전달하게 해준다.

---
TLS는 하나의 단순한 프로토콜이 아니라, 크게 두 계층으로 나뉜다.

![](../images/Pasted%20image%2020260519180509.png)

**TLS Handshake Protocol**은 통신을 시작하기 전에 보안 파라미터를 협상하는 단계이다. 즉, 어떤 TLS 버전을 쓸지, 어떤 암호 알고리즘을 쓸지, 서버 인증서는 유효한지, 세션 키는 어떻게 만들지 등을 정한다.

**TLS Record Protocol**은 실제 데이터를 안전하게 나누고, 암호화하고, 무결성을 붙여 전송하는 계층이다.

쉽게 말하면 다음과 같다.

- Handshake Protocol = 보안 통신을 시작하기 위한 준비 과정
- Record Protocol = 준비된 키로 실제 데이터를 안전하게 보내는 과정

즉, TLS는 처음부터 데이터를 바로 암호화해서 보내는 것이 아니라, 먼저 핸드셰이크를 통해 서로 사용할 보안 설정과 키를 정한 뒤, Record 계층에서 실제 데이터를 처리한다.

---
# Connection and Session
TLS에서는 **Connection**과 **Session**을 구분한다.

Connection은 실제 클라이언트와 서버 사이의 일시적인 통신 연결이다.  
즉, TCP 연결 하나와 비슷하게 생각하면 된다.

Session은 클라이언트와 서버가 공유하는 보안 설정의 묶음이다. Handshake Protocol을 통해 만들어지며, 암호 알고리즘, master secret 같은 정보를 포함한다.
중요한 점은 하나의 Session이 여러 Connection에서 재사용될 수 있다는 것이다.

예를 들어 사용자가 같은 서버에 여러 번 접속할 때, 매번 처음부터 모든 핸드셰이크를 다시 수행하면 비용이 크다. 그래서 이전에 만든 Session을 재사용하면 더 빠르게 보안 연결을 만들 수 있다.

정리하면 다음과 같다.

- Session = 보안 설정의 묶음
- Connection = 실제 통신 링크

Session은 비교적 논리적인 개념이고, Connection은 실제 데이터가 오가는 연결이라고 보면 된다.

---
# Session State
Session State는 TLS 세션이 유지하기 위해 필요한 정보들이다.

- **Session Identifier**
	서버가 특정 세션을 식별하기 위해 선택한 바이트 문자열이다. 나중에 클라이언트가 같은 세션을 재사용하고 싶을 때 이 값을 사용할 수 있다.

- **Peer Certificate**
	상대방의 X.509 v3 인증서이다. 보통 웹에서는 서버 인증서가 여기에 해당한다.

- **Compression Method**
	TLS에서는 암호화하기 전에 데이터를 압축하는 방식이 정의되어 있었지만, 실제 보안 문제 때문에 현대 TLS에서는 압축을 거의 사용하지 않는다.

- **Cipher Spec**
	사용할 암호 알고리즘들의 집합이다. 예를 들어 키 교환 방식, 대칭키 암호 방식, MAC 또는 AEAD 방식 등을 포함한다.

즉, Session State는 “이 세션에서 어떤 보안 설정을 사용할 것인가?”에 대한 정보를 저장한다.

##### Master Secret
Session State에는 **Master Secret**도 포함된다. Master Secret은 클라이언트와 서버가 공유하는 48바이트 비밀값이다. 이 값은 직접 데이터를 암호화하는 키라기보다는, 이후 여러 실제 키를 만들어내는 핵심 재료라고 보면 된다.

TLS에서는 하나의 키만 사용하는 것이 아니라, 방향별로 여러 키를 만든다.

예를 들어 다음과 같은 키들이 필요하다.

- 클라이언트가 보낼 때 쓰는 MAC key
- 서버가 보낼 때 쓰는 MAC key
- 클라이언트가 보낼 때 쓰는 encryption key
- 서버가 보낼 때 쓰는 encryption key
- IV

이런 키들을 만들기 위한 중심 재료가 Master Secret이다.

##### Is Resumable
또한 **Is Resumable**이라는 값이 있다. 
이 세션을 나중에 다시 사용할 수 있는지 나타내는 플래그이다.

---
# Connection State
Connection State는 실제 연결 단위에서 필요한 정보이다. Session State가 비교적 장기적인 보안 설정이라면, Connection State는 현재 연결에서 실제로 데이터를 주고받을 때 필요한 상태값이다.

여기에는 **server random**과 **client random**이 있다. 
클라이언트와 서버가 각각 만든 랜덤 값이며, 키 생성 과정에서 사용된다.

또한 방향별 MAC key가 있다.

- Server write MAC key
- Client write MAC key

Server write MAC key는 서버가 보내는 데이터의 MAC을 만들 때 사용한다.  
Client write MAC key는 클라이언트가 보내는 데이터의 MAC을 만들 때 사용한다.

그리고 방향별 암호화 키도 있다.

- Server write key
- Client write key

Server write key는 서버가 데이터를 암호화할 때 쓰고, 클라이언트가 복호화할 때 쓴다.  
Client write key는 클라이언트가 데이터를 암호화할 때 쓰고, 서버가 복호화할 때 쓴다.

즉, TLS는 양방향 통신이기 때문에 클라이언트→서버, 서버→클라이언트 방향의 키를 따로 둔다.

---

Connection State에는 IV와 sequence number도 포함된다.

**IV**, 즉 Initialization Vector는 블록 암호 모드에서 초기 상태를 제공하는 값이다.  
CBC 같은 모드에서는 이전 암호문 블록이 다음 레코드의 IV처럼 사용되기도 한다.

**Sequence Number**는 각 방향별 메시지 번호이다.  
클라이언트가 보내는 메시지와 서버가 보내는 메시지에 각각 별도의 sequence number가 있다.이 sequence number는 무결성 검증에서 중요하다. 왜냐하면 공격자가 메시지를 재전송하거나 순서를 바꾸는 것을 막는 데 도움이 되기 때문이다.

예를 들어 중간 공격자가 예전 TLS 레코드를 다시 보내는 replay attack을 시도할 수 있다. 이때 sequence number가 MAC 계산에 포함되면, 같은 메시지를 단순히 다시 보내도 올바른 순서의 메시지로 인정되지 않는다.
