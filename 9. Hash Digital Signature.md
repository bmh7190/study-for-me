
# Hash Function
해쉬 함수는 어떤 값 x를 입력 받아서 고정된 길이의 출력값 `H(x)` 를 만드는 함수이다.

해쉬 함수는 두 가지 성질을 가진다.

#### 1. Compression
입력의 길이가 얼마든지 될 수 있다. 짧은 문자열일 수도 있고, 긴 문자열일 수도 있는데, 항상 고정된 길이로 나온다.

#### 2. Ease of computation
입력 x가 주어졌을 때 `H(x)`를 계산하는 것은 쉬워야 한다.

# Hash Function & MAC

![](../images/Pasted%20image%2020260517175102.png)

일반 해시 함수는 message 를 Hash Function에 넣으면 고정 길이인 Hash value 가 나온다. 여기에는 비밀키가 없다.

반면 MAC 은 message 와 secret key를 더해서 MAC function 안에 넣어서 MAC value 가 나온다. MAC 은 메세지 뿐만 아니라 비밀키도 같이 사용한다. 그래서 키를 모르는 사람은 올바른 MAC 값을 만들 수 없다.

Hash Function 과 MAC 의 중요한 차이는 인증이 가능한가 아닌가 이다. Hash 같은 경우에는 무결성 확인이 가능하지만 누가 만들었는지 인증은 할 수 없다. 하지만 MAC 은 중간에 key를 사용하므로 무결성과 인증을 둘 다 할 수 있다.

---
# Hash Function의 목적
해시 함수 H는 가변 길이 데이터 M을 입력 받고 고정 길이 해시값 h를 만든다.

`h = H(M)` 

해시 함수의 가장 큰 목적은 Data integrity, 데이터 무결성이다.
예를 들어 어떤 파일을 다운로드 했을 때, 제공자가 SHA-256 해시값을 같이 올려두는 경우가 있다. 사용자는 파일을 받은 후 직접 해시를 계산해서 제공된 해시값과 비교한다.  같으면 파일이 변조되지 않았다고 판단할 수 있다.

그런데 암호학적인 해시 함수는 그냥 요약만 하면 안되고, 두 가지 조건을 만족해야 한다.

#### One way property 일방향성
첫 번째, 특정 해시값이 주어졌을 때, 그 해시 값을 만드는 입력을 찾기 어려워야 한다.

#### collision free property 충돌저항성
두 번재, 서로 다른 두 입력이 같은 해시값을 만들도록 찾기 어려워야 한다.

# Checking Integrity

![](../images/Pasted%20image%2020260517175639.png)

해시를 이용해서 무결성을 확인하는 방법은 간단하다. 먼저 원본 데이터의 해시 값을 저장해둔다. 예시에서는 previous digest이다. 그리고 나중에 데이터를 받는 사람은 현재 데이터의 해시값을 다시 계산한다. 이것을 Current digest 라고 하고, 두 값을 비교한다.

값이 같으면 데이터가 바뀌지 않은 것이고, 다르면 데이터가 변경된 것이다.

# Attack Against Hash Function

![](../images/Pasted%20image%2020260517175919.png)

해시 함수 공격은 크게 3가지로 나뉜다. 

첫 번째는 Preimage attack이다.
해시  h가 주어졌을 때, `H(x) = h` 가 되는 입력 x를 찾는 공격이다.

두 번째는 Second preimage attack이다.
어떤 원본 메세지 x가 주어졌을 때, `H(y) = H(x)` 가 되는 다른 메세지 y를 찾는 공격이다.

세 번째는 Collison attack 이다.
공격자가 아무 두 메세지 x, y를 골라서 `x ≠ y`인데 `H(x) = H(y)`가 되게 찾는 공격이다.

이 중에서는 collision attack이 상대적으로 더 쉽다.

# Use of a Hash Function
해시 함수는 메시지나 파일의 무결성을 확인할 때 사용된다. 보낸 사람이 메시지와 해시값을 함께 보낸다고 가정한다. 받는 사람은 받은 메시지를 다시 해시해서 전달받은 해시값과 비교한다. 값이 같으면 메시지가 전송 중 변경되지 않았다고 판단한다.

![](../images/Pasted%20image%2020260518230759.png)

하지만 이 구조만으로는 공격자를 완전히 막을 수 없다. 왜냐하면 공격자가 메시지를 바꾼 뒤, 바뀐 메시지에 대한 해시값도 새로 계산해서 보내면 받는 사람은 이를 구분하지 못할 수 있기 때문이다.

따라서 해시 함수는 기본적으로 **무결성 확인의 재료**이고, 인증을 위해서는 비밀키를 사용하는 MAC이나 개인키를 사용하는 전자서명과 결합해야 한다.

# Other Hash Function Uses
해시 함수는 여러 곳에 사용된다.

##### **비밀번호 저장**
비밀번호를 그대로 저장하지 않고 비밀번호의 해시값을 저장한다. 사용자가 로그인할 때 입력한 비밀번호를 다시 해시해서 저장된 해시값과 비교한다.

##### **침입 탐지와 바이러스 탐지**  
시스템 파일의 해시값을 미리 저장해 두고, 나중에 파일을 다시 해시해서 값이 달라졌는지 확인한다. 해시값이 달라졌다면 파일이 변조되었을 가능성이 있다.

##### **PRF나 PRNG 구성**  
해시 함수는 입력을 랜덤해 보이는 출력으로 바꾸기 때문에, 의사난수 생성이나 대칭키 생성 과정에도 활용될 수 있다. 자료에서도 해시 함수가 비밀번호 파일, 침입 및 바이러스 탐지, PRF/PRNG와 대칭키 생성에 활용될 수 있다고 정리한다.

# Terminology
여기서는 해시 함수에서 자주 나오는 용어를 정리한다.

#### 1. Preimage
**Preimage**는 어떤 해시값을 만들어낸 입력값이다.

`h = H(x)`라면, `x`는 `h`의 preimage이다.

해시 함수는 many-to-one mapping이다. 즉, 입력 공간은 매우 크고 출력 공간은 고정되어 있으므로, 이론적으로는 하나의 해시값에 여러 입력이 대응될 수 있다.

#### 2. Collision
**Collision**은 서로 다른 두 입력이 같은 해시값을 갖는 경우이다.

`x ≠ y` 이고 `H(x) = H(y)`이면 collision이다.

해시 함수는 출력 길이가 고정되어 있기 때문에 충돌은 이론적으로 반드시 존재한다. 다만 중요한 것은 충돌의 존재 여부가 아니라, 공격자가 충돌을 현실적으로 찾을 수 있느냐이다.

암호학적 해시 함수에서는 충돌을 찾는 것이 계산적으로 불가능에 가까워야 한다.

# Requirements and Security

암호학적 해시 함수가 만족해야 하는 요구사항은 다음과 같이 정리할 수 있다.

- 먼저 입력 크기는 가변적이어야 한다. 어떤 크기의 데이터에도 적용할 수 있어야 한다.

- 출력 크기는 고정되어야 한다. 입력이 아무리 길어도 해시값의 길이는 일정해야 한다.

- 계산은 효율적이어야 한다. 주어진 입력에 대해 `H(x)`를 계산하는 것은 쉬워야 한다.

보안 성질은 세 가지가 중요하다.

- **preimage resistance**  
	해시값 `h`가 주어졌을 때, `H(y) = h`가 되는 입력 `y`를 찾기 어려워야 한다.

- **second preimage resistance**
	입력 `x`가 주어졌을 때, `x`와 다르면서 같은 해시값을 갖는 `y`를 찾기 어려워야 한다.

- **collision resistance**
	서로 다른 두 입력 `x`, `y`를 찾아서 `H(x) = H(y)`가 되게 만드는 것이 어려워야 한다.

여기에 추가로 해시 출력은 랜덤해 보여야 한다. 즉, 출력이 특정 패턴을 가지면 공격에 활용될 수 있기 때문에 pseudo-randomness도 필요하다.

# Birthday Problems
Birthday problem은 해시 함수의 공격 난이도를 이해하기 위해 사용된다.
서로 생일을 모른다고 했을 때, 생일 자체가 Hash 값으로 생각해보자

여기서 세 가지 상황을 비교한다.

##### “정해진 생일을 가진 학생이 있을 확률”  
예를 들어 1월 1일 생일인 학생을 찾는 것이다. 이건 특정 해시값에 대응되는 입력을 찾는 preimage attack과 비슷하다.

##### “교수가 선택한 학생과 같은 생일을 가진 학생이 있을 확률”
이미 특정 학생이 정해져 있고, 그 학생과 같은 생일을 가진 다른 학생을 찾는 것이다. 이건 second preimage attack과 비슷하다.

##### “학생들 중 아무 두 명이나 생일이 같을 확률”  
이건 collision attack과 비슷하다.

중요한 점은 세 번째가 훨씬 쉽다는 것이다. 정해진 생일을 맞히는 것보다, 아무 두 사람이 같은 생일이 되는 경우를 찾는 것이 훨씬 빨리 발생한다.

---

![](../images/Pasted%20image%2020260518231825.png)

이 표에서 가장 중요한 결론은 collision attack의 난이도가 preimage attack보다 훨씬 낮다는 것이다. 가능한 해시값의 개수를 `N`이라고 하면, preimage나 second preimage는 대략 `N`에 비례하는 시도가 필요하다. 반면 collision은 대략 `√N` 정도의 시도만으로도 가능성이 커진다. 그래서 해시 출력이 `n`비트이면 가능한 해시값 개수는 `2^n`개이다.  

preimage 공격은 대략 `2^n` 수준이 필요하지만, collision 공격은 birthday paradox 때문에 대략 `2^(n/2)` 수준으로 가능하다. 예를 들어 256비트 해시 함수의 충돌 저항성은 대략 128비트 수준으로 본다.

##### Preimage
정해진 생일이 하나 있다고 하자. 예를 들어 “1월 1일 생일인 학생이 있는가?”를 확인하는 것이다.

이 경우 50% 정도의 확률을 얻으려면 학생 수가 상당히 많이 필요하다. 자료에서는 `N = 365`일 때 약 253명이 필요하다고 정리한다. 해시 함수로 보면 특정 해시값 하나가 정해져 있고, 그 해시값을 만들어내는 입력을 찾는 상황이다. 따라서 특정 목표값을 맞히는 것은 어렵다. 이것이 preimage resistance와 연결된다.

##### Collision
이번에는 정해진 생일이 없다. 단지 학생들 중 아무 두 명이나 생일이 같으면 된다.

이 경우 `N = 365`일 때 약 23명만 있어도 같은 생일을 가진 두 사람이 있을 확률이 50% 정도가 된다. 이게 birthday paradox이다.

해시 함수에서도 마찬가지이다. 특정 해시값을 맞히는 것은 어렵지만, 서로 같은 해시값을 갖는 아무 두 입력을 찾는 것은 훨씬 쉽다. 그래서 해시 함수의 출력 길이는 충돌 공격을 고려해서 충분히 길어야 한다.

# Collision Resistant Attacks
collision resistant attack은 공격자가 같은 해시값을 갖는 서로 다른 두 메시지를 찾으려는 공격이다. 이 공격의 난이도는 birthday paradox로 설명된다. 예를 들어 120비트 해시 함수가 있다고 하면, 전체 출력 공간은 2<sup>120</sup> 이지만 충돌 공격은 대략 2<sup>60</sup>정도의 작업량으로 가능해진다.

자료에서는 1GHz 컴퓨터 기준으로 2<sup>60</sup> CPU cycle이 약 35년 정도 걸리고, RTX 4090 GPU의 SHA-1 처리량을 기준으로 하면 약 264일 정도로 계산된다고 설명한다.

여기서 핵심은 해시 출력 길이가 짧으면 충돌 공격이 현실적인 문제가 될 수 있다는 점이다. 따라서 현대적인 해시 함수에서는 SHA-256처럼 충분히 긴 출력 길이를 사용한다.

# General Construction (Merkle-Damgard)
Merkle-Damgard 구조는 해시 함수를 구성하는 일반적인 방식이다. 긴 메시지를 한 번에 처리하는 것이 아니라, 일정한 크기의 블록으로 나누고 각 블록을 순차적으로 압축 함수에 넣는다. 

![](../images/Pasted%20image%2020260518233843.png)

먼저 메시지에 padding과 length encoding을 적용한다. 그다음 메시지를 M<sub>1</sub>, M<sub>2</sub>, ..., M<sub>t</sub>와 같은 블록으로 나눈다. 초기값 `IV`에서 시작해서 다음처럼 chaining variable을 갱신한다.

`H1 = f(H0, M1)`  
`H2 = f(H1, M2)`  
`H3 = f(H2, M3)`

이런 식으로 마지막 블록까지 처리한다. 최종적으로 나온 값이 해시값이 된다. 필요하면 마지막에 output transformation을 적용할 수도 있다. 즉, Merkle-Damgard 구조는 “이전 블록까지의 결과와 현재 메시지 블록을 압축해서 다음 상태를 만드는 구조”라고 정리할 수 있다.

# Secure Hash Algorithm (SHA)
SHA는 Secure Hash Algorithm의 약자이다. SHA는 NIST에서 설계한 해시 함수 계열이다. 처음에는 1993년에 발표되었고, 이후 SHA-1로 개정되었다.

- SHA-1은 160비트 해시값을 생성한다.
- SHA-2는 더 긴 출력 길이를 제공하는 계열이다.  
- SHA-3는 SHA-2와 다른 구조를 사용하는 새로운 표준이다.

## SHA 3
SHA-1은 실제 충돌이 공개되면서 안전하지 않은 것으로 평가된다. 자료에서도 2017년에 Google과 CWI가 SHA-1 충돌을 공개했다고 설명한다. SHA-2는 아직 널리 사용되지만, SHA-1과 같은 계열의 구조를 공유한다는 점에서 장기적으로 다른 구조의 해시 함수가 필요했다. 그래서 NIST는 SHA-3 경쟁을 열었고, 2012년에 Keccak이 SHA-3로 선정되었다.

SHA-3는 SHA-2를 단순히 대체한다기보다, SHA-2와 함께 사용할 수 있는 새로운 해시 표준이다.
SHA-3의 중요한 특징은 Merkle-Damgard 구조가 아니라 **sponge construction**을 사용한다는 점이다.

![](../images/Pasted%20image%2020260518234231.png)

SHA-3는 sponge construction을 사용한다. 스펀지 구조는 크게 두 단계로 볼 수 있다.

첫 번째는 **absorb 단계**이다.  
메시지를 내부 상태에 흡수한다. 메시지 블록을 내부 상태에 XOR하고, permutation을 적용하면서 상태를 갱신한다.

두 번째는 **squeeze 단계**이다.  
내부 상태로부터 필요한 길이만큼 출력을 뽑아낸다.

즉, SHA-3는 입력을 내부 상태에 흡수한 뒤, 그 상태에서 해시값을 짜내는 구조라고 이해하면 된다. Merkle-Damgard 방식이 “블록을 순차적으로 압축하는 구조”라면, SHA-3는 “내부 상태에 데이터를 흡수하고 출력값을 생성하는 구조”이다.

# MAC 
MAC은 Message Authentication Code이다. MAC은 비밀키를 공유하는 두 당사자 사이에서 메시지의 무결성과 인증을 확인하기 위해 사용된다.

1. 송신자는 메시지와 비밀키를 이용해 MAC 값을 만든다.
	`MAC = MAC_K(M)`
2. 수신자는 같은 비밀키로 다시 MAC 값을 계산하고, 전달받은 MAC 값과 비교한다.
3. 값이 같으면 메시지가 바뀌지 않았고, 비밀키를 아는 사람이 만든 메시지라고 판단할 수 있다.

즉, MAC은 **무결성 + 인증**을 제공한다. 하지만 MAC은 부인방지를 제공하지 못한다. 왜냐하면 송신자와 수신자가 같은 비밀키를 공유하기 때문에, 제3자 입장에서는 둘 중 누가 MAC을 만들었는지 구분하기 어렵기 때문이다.

# HMAC
HMAC은 Hash-based MAC으로 암호학적 해시 함수를 기반으로 MAC을 구성하는 방식이다.

HMAC을 사용하는 이유는 해시 함수가 소프트웨어에서 빠르게 동작하고, MD5나 SHA 같은 해시 함수 라이브러리가 널리 제공되기 때문이다.

개념적으로 HMAC은 “비밀키와 메시지를 함께 사용해서 해시 기반 인증값을 만드는 방식”이다.

단순히 `H(key || message)`처럼 붙이는 방식은 안전하지 않을 수 있기 때문에, 실제 HMAC은 내부 패드와 외부 패드를 사용하여 구성된다.

![](../images/Pasted%20image%2020260518234852.png)

정리하면 HMAC은 해시 함수의 빠른 계산성과 MAC의 인증 기능을 결합한 방식이다.

# Digital Signature
전자서명은 MAC과 비슷하게 메시지 인증을 제공하지만, 사용하는 키 구조가 다르다.

- MAC은 **대칭키** 기반이다.  
- 전자서명은 **비대칭키** 기반이다.

전자서명에서는 송신자가 자신의 **개인키**로 서명하고, 검증자는 송신자의 **공개키**로 검증한다.

일반적인 흐름은 다음과 같다.

1. 메시지를 해시한다.  
2. 해시값에 대해 개인키로 서명한다.  
3. 메시지와 서명을 함께 보낸다.  
4. 검증자는 메시지를 다시 해시하고, 공개키로 서명을 검증한다.

이 구조를 사용하면 메시지가 바뀌었는지 확인할 수 있고, 해당 개인키 소유자가 서명했다는 것도 확인할 수 있다. 또한 공개키로 누구나 검증할 수 있으므로, 제3자에게도 증명 가능하다. 이것이 부인방지로 이어진다.

# Use of Digital Signature
전자서명은 보통 메시지 전체를 직접 서명하지 않는다. 메시지를 먼저 해시한 뒤, 그 해시값에 서명한다.

이유는 두 가지이다.

- 공개키 연산은 느리다. 메시지 전체에 대해 직접 서명하면 비효율적이다.
- 해시값은 고정 길이이다. 메시지 크기와 관계없이 일정한 크기의 해시값만 서명하면 되므로 처리하기 좋다.

![](../images/Pasted%20image%2020260518235303.png)

검증자는 받은 메시지를 다시 해시하고, 서명을 공개키로 검증해서 나온 값과 비교한다. 두 값이 같으면 메시지가 서명 이후 변경되지 않았고, 서명자의 개인키로 만들어진 서명임을 확인할 수 있다.
