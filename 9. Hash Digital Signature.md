
# Hash Function
해쉬 함수는 어떤 값 x를 입력 받아서 고정된 길이의 출력값 `H(x)` 를 만드는 함수이다.

해쉬 함수는 두 가지 성질을 가진다.

#### 1. Compression
입력의 길이가 얼마든지 될 수 있다. 짧은 문자열일 수도 있고, 긴 문자열일 수도 있는데, 항상 고정된 길이로 나온다.

#### 2. Ease of computation
입력 x가 주어졌을 때 `H(x)`를 계산하는 것은 쉬워야 한다.

# Hash Function & MAC

![](../images/Pasted%20image%2020260517175102.png)

일반 해시 함수는 message 를 Hash Function에 넣으면 고정 길이인 Hash value 가 나온다. 여기에는 비밀키가 없다.

반면 MAC 은 message 와 secret key를 더해서 MAC function 안에 넣어서 MAC value 가 나온다. MAC 은 메세지 뿐만 아니라 비밀키도 같이 사용한다. 그래서 키를 모르는 사람은 올바른 MAC 값을 만들 수 없다.

Hash Function 과 MAC 의 중요한 차이는 인증이 가능한가 아닌가 이다. Hash 같은 경우에는 무결성 확인이 가능하지만 누가 만들었는지 인증은 할 수 없다. 하지만 MAC 은 중간에 key를 사용하므로 무결성과 인증을 둘 다 할 수 있다.

---
# Hash Function의 목적
해시 함수 H는 가변 길이 데이터 M을 입력 받고 고정 길이 해시값 h를 만든다.

`h = H(M)` 

해시 함수의 가장 큰 목적은 Data integrity, 데이터 무결성이다.
예를 들어 어떤 파일을 다운로드 했을 때, 제공자가 SHA-256 해시값을 같이 올려두는 경우가 있다. 사용자는 파일을 받은 후 직접 해시를 계산해서 제공된 해시값과 비교한다.  같으면 파일이 변조되지 않았다고 판단할 수 있다.

그런데 암호학적인 해시 함수는 그냥 요약만 하면 안되고, 두 가지 조건을 만족해야 한다.

#### One way property 일방향성
첫 번째, 특정 해시값이 주어졌을 때, 그 해시 값을 만드는 입력을 찾기 어려워야 한다.

#### collision free property 충돌저항성
두 번재, 서로 다른 두 입력이 같은 해시값을 만들도록 찾기 어려워야 한다.

# Checking Integrity

![](../images/Pasted%20image%2020260517175639.png)

해시를 이용해서 무결성을 확인하는 방법은 간단하다. 먼저 원본 데이터의 해시 값을 저장해둔다. 예시에서는 previous digest이다. 그리고 나중에 데이터를 받는 사람은 현재 데이터의 해시값을 다시 계산한다. 이것을 Current digest 라고 하고, 두 값을 비교한다.

값이 같으면 데이터가 바뀌지 않은 것이고, 다르면 데이터가 변경된 것이다.

# Attack Against Hash Function

![](../images/Pasted%20image%2020260517175919.png)

해시 함수 공격은 크게 3가지로 나뉜다. 

첫 번째는 Preimage attack이다.
해시  h가 주어졌을 때, `H(x) = h` 가 되는 입력 x를 찾는 공격이다.

두 번째는 Second preimage attack이다.
어떤 원본 메세지 x가 주어졌을 때, `H(y) = H(x)` 가 되는 다른 메세지 y를 찾는 공격이다.

세 번째는 Collison attack 이다.
공격자가 아무 두 메세지 x, y를 골라서 `x ≠ y`인데 `H(x) = H(y)`가 되게 찾는 공격이다.

이 중에서는 collision attack이 상대적으로 더 쉽다.

# Use of a Hash Function
해시 함수는 메시지나 파일의 무결성을 확인할 때 사용된다. 보낸 사람이 메시지와 해시값을 함께 보낸다고 가정한다. 받는 사람은 받은 메시지를 다시 해시해서 전달받은 해시값과 비교한다. 값이 같으면 메시지가 전송 중 변경되지 않았다고 판단한다.

![](../images/Pasted%20image%2020260518230759.png)

하지만 이 구조만으로는 공격자를 완전히 막을 수 없다. 왜냐하면 공격자가 메시지를 바꾼 뒤, 바뀐 메시지에 대한 해시값도 새로 계산해서 보내면 받는 사람은 이를 구분하지 못할 수 있기 때문이다.

따라서 해시 함수는 기본적으로 **무결성 확인의 재료**이고, 인증을 위해서는 비밀키를 사용하는 MAC이나 개인키를 사용하는 전자서명과 결합해야 한다.

# Other Hash Function Uses
해시 함수는 여러 곳에 사용된다.

##### **비밀번호 저장**
비밀번호를 그대로 저장하지 않고 비밀번호의 해시값을 저장한다. 사용자가 로그인할 때 입력한 비밀번호를 다시 해시해서 저장된 해시값과 비교한다.

##### **침입 탐지와 바이러스 탐지**  
시스템 파일의 해시값을 미리 저장해 두고, 나중에 파일을 다시 해시해서 값이 달라졌는지 확인한다. 해시값이 달라졌다면 파일이 변조되었을 가능성이 있다.

##### **PRF나 PRNG 구성**  
해시 함수는 입력을 랜덤해 보이는 출력으로 바꾸기 때문에, 의사난수 생성이나 대칭키 생성 과정에도 활용될 수 있다. 자료에서도 해시 함수가 비밀번호 파일, 침입 및 바이러스 탐지, PRF/PRNG와 대칭키 생성에 활용될 수 있다고 정리한다.

# Terminology
여기서는 해시 함수에서 자주 나오는 용어를 정리한다.

#### 1. Preimage
**Preimage**는 어떤 해시값을 만들어낸 입력값이다.

`h = H(x)`라면, `x`는 `h`의 preimage이다.

해시 함수는 many-to-one mapping이다. 즉, 입력 공간은 매우 크고 출력 공간은 고정되어 있으므로, 이론적으로는 하나의 해시값에 여러 입력이 대응될 수 있다.

#### 2. Collision
**Collision**은 서로 다른 두 입력이 같은 해시값을 갖는 경우이다.

`x ≠ y` 이고 `H(x) = H(y)`이면 collision이다.

해시 함수는 출력 길이가 고정되어 있기 때문에 충돌은 이론적으로 반드시 존재한다. 다만 중요한 것은 충돌의 존재 여부가 아니라, 공격자가 충돌을 현실적으로 찾을 수 있느냐이다.

암호학적 해시 함수에서는 충돌을 찾는 것이 계산적으로 불가능에 가까워야 한다.

# Requirements and Security

암호학적 해시 함수가 만족해야 하는 요구사항은 다음과 같이 정리할 수 있다.

- 먼저 입력 크기는 가변적이어야 한다. 어떤 크기의 데이터에도 적용할 수 있어야 한다.

- 출력 크기는 고정되어야 한다. 입력이 아무리 길어도 해시값의 길이는 일정해야 한다.

- 계산은 효율적이어야 한다. 주어진 입력에 대해 `H(x)`를 계산하는 것은 쉬워야 한다.

보안 성질은 세 가지가 중요하다.

- **preimage resistance**  
	해시값 `h`가 주어졌을 때, `H(y) = h`가 되는 입력 `y`를 찾기 어려워야 한다.

- **second preimage resistance**
	입력 `x`가 주어졌을 때, `x`와 다르면서 같은 해시값을 갖는 `y`를 찾기 어려워야 한다.

- **collision resistance**
	서로 다른 두 입력 `x`, `y`를 찾아서 `H(x) = H(y)`가 되게 만드는 것이 어려워야 한다.

여기에 추가로 해시 출력은 랜덤해 보여야 한다. 즉, 출력이 특정 패턴을 가지면 공격에 활용될 수 있기 때문에 pseudo-randomness도 필요하다.

# Birthday Problems
Birthday problem은 해시 함수의 공격 난이도를 이해하기 위해 사용된다.
서로 생일을 모른다고 했을 때, 생일 자체가 Hash 값으로 생각해보자

여기서 세 가지 상황을 비교한다.

##### “정해진 생일을 가진 학생이 있을 확률”  
예를 들어 1월 1일 생일인 학생을 찾는 것이다. 이건 특정 해시값에 대응되는 입력을 찾는 preimage attack과 비슷하다.

##### “교수가 선택한 학생과 같은 생일을 가진 학생이 있을 확률”
이미 특정 학생이 정해져 있고, 그 학생과 같은 생일을 가진 다른 학생을 찾는 것이다. 이건 second preimage attack과 비슷하다.

##### “학생들 중 아무 두 명이나 생일이 같을 확률”  
이건 collision attack과 비슷하다.

중요한 점은 세 번째가 훨씬 쉽다는 것이다. 정해진 생일을 맞히는 것보다, 아무 두 사람이 같은 생일이 되는 경우를 찾는 것이 훨씬 빨리 발생한다.

---

![](../images/Pasted%20image%2020260518231825.png)

이 표에서 가장 중요한 결론은 collision attack의 난이도가 preimage attack보다 훨씬 낮다는 것이다. 가능한 해시값의 개수를 `N`이라고 하면, preimage나 second preimage는 대략 `N`에 비례하는 시도가 필요하다. 반면 collision은 대략 `√N` 정도의 시도만으로도 가능성이 커진다. 그래서 해시 출력이 `n`비트이면 가능한 해시값 개수는 `2^n`개이다.  

preimage 공격은 대략 `2^n` 수준이 필요하지만, collision 공격은 birthday paradox 때문에 대략 `2^(n/2)` 수준으로 가능하다. 예를 들어 256비트 해시 함수의 충돌 저항성은 대략 128비트 수준으로 본다.

##### Preimage
정해진 생일이 하나 있다고 하자. 예를 들어 “1월 1일 생일인 학생이 있는가?”를 확인하는 것이다.

이 경우 50% 정도의 확률을 얻으려면 학생 수가 상당히 많이 필요하다. 자료에서는 `N = 365`일 때 약 253명이 필요하다고 정리한다. 해시 함수로 보면 특정 해시값 하나가 정해져 있고, 그 해시값을 만들어내는 입력을 찾는 상황이다. 따라서 특정 목표값을 맞히는 것은 어렵다. 이것이 preimage resistance와 연결된다.

##### Collision
이번에는 정해진 생일이 없다. 단지 학생들 중 아무 두 명이나 생일이 같으면 된다.

이 경우 `N = 365`일 때 약 23명만 있어도 같은 생일을 가진 두 사람이 있을 확률이 50% 정도가 된다. 이게 birthday paradox이다.

해시 함수에서도 마찬가지이다. 특정 해시값을 맞히는 것은 어렵지만, 서로 같은 해시값을 갖는 아무 두 입력을 찾는 것은 훨씬 쉽다. 그래서 해시 함수의 출력 길이는 충돌 공격을 고려해서 충분히 길어야 한다.

# Collision Resistant Attacks
collision resistant attack은 공격자가 같은 해시값을 갖는 서로 다른 두 메시지를 찾으려는 공격이다. 이 공격의 난이도는 birthday paradox로 설명된다. 예를 들어 120비트 해시 함수가 있다고 하면, 전체 출력 공간은 2<sup>120</sup> 이지만 충돌 공격은 대략 2<sup>60</sup>정도의 작업량으로 가능해진다.

자료에서는 1GHz 컴퓨터 기준으로 2<sup>60</sup> CPU cycle이 약 35년 정도 걸리고, RTX 4090 GPU의 SHA-1 처리량을 기준으로 하면 약 264일 정도로 계산된다고 설명한다.

여기서 핵심은 해시 출력 길이가 짧으면 충돌 공격이 현실적인 문제가 될 수 있다는 점이다. 따라서 현대적인 해시 함수에서는 SHA-256처럼 충분히 긴 출력 길이를 사용한다.

# General Construction (Merkle-Damgard)
Merkle-Damgard 구조는 해시 함수를 구성하는 일반적인 방식이다. 긴 메시지를 한 번에 처리하는 것이 아니라, 일정한 크기의 블록으로 나누고 각 블록을 순차적으로 압축 함수에 넣는다. 

![](../images/Pasted%20image%2020260518233843.png)

먼저 메시지에 padding과 length encoding을 적용한다. 그다음 메시지를 M<sub>1</sub>, M<sub>2</sub>, ..., M<sub>t</sub>와 같은 블록으로 나눈다. 초기값 `IV`에서 시작해서 다음처럼 chaining variable을 갱신한다.

`H1 = f(H0, M1)`  
`H2 = f(H1, M2)`  
`H3 = f(H2, M3)`

이런 식으로 마지막 블록까지 처리한다. 최종적으로 나온 값이 해시값이 된다. 필요하면 마지막에 output transformation을 적용할 수도 있다. 즉, Merkle-Damgard 구조는 “이전 블록까지의 결과와 현재 메시지 블록을 압축해서 다음 상태를 만드는 구조”라고 정리할 수 있다.

# Secure Hash Algorithm (SHA)
SHA는 Secure Hash Algorithm의 약자이다. SHA는 NIST에서 설계한 해시 함수 계열이다. 처음에는 1993년에 발표되었고, 이후 SHA-1로 개정되었다.

- SHA-1은 160비트 해시값을 생성한다.
- SHA-2는 더 긴 출력 길이를 제공하는 계열이다.  
- SHA-3는 SHA-2와 다른 구조를 사용하는 새로운 표준이다.

## SHA 3
SHA-1은 실제 충돌이 공개되면서 안전하지 않은 것으로 평가된다. 자료에서도 2017년에 Google과 CWI가 SHA-1 충돌을 공개했다고 설명한다. SHA-2는 아직 널리 사용되지만, SHA-1과 같은 계열의 구조를 공유한다는 점에서 장기적으로 다른 구조의 해시 함수가 필요했다. 그래서 NIST는 SHA-3 경쟁을 열었고, 2012년에 Keccak이 SHA-3로 선정되었다.

SHA-3는 SHA-2를 단순히 대체한다기보다, SHA-2와 함께 사용할 수 있는 새로운 해시 표준이다.
SHA-3의 중요한 특징은 Merkle-Damgard 구조가 아니라 **sponge construction**을 사용한다는 점이다.

![](../images/Pasted%20image%2020260518234231.png)

SHA-3는 sponge construction을 사용한다. 스펀지 구조는 크게 두 단계로 볼 수 있다.

첫 번째는 **absorb 단계**이다.  
메시지를 내부 상태에 흡수한다. 메시지 블록을 내부 상태에 XOR하고, permutation을 적용하면서 상태를 갱신한다.

두 번째는 **squeeze 단계**이다.  
내부 상태로부터 필요한 길이만큼 출력을 뽑아낸다.

즉, SHA-3는 입력을 내부 상태에 흡수한 뒤, 그 상태에서 해시값을 짜내는 구조라고 이해하면 된다. Merkle-Damgard 방식이 “블록을 순차적으로 압축하는 구조”라면, SHA-3는 “내부 상태에 데이터를 흡수하고 출력값을 생성하는 구조”이다.

# MAC 
MAC은 Message Authentication Code이다. MAC은 비밀키를 공유하는 두 당사자 사이에서 메시지의 무결성과 인증을 확인하기 위해 사용된다.

1. 송신자는 메시지와 비밀키를 이용해 MAC 값을 만든다.
	`MAC = MAC_K(M)`
2. 수신자는 같은 비밀키로 다시 MAC 값을 계산하고, 전달받은 MAC 값과 비교한다.
3. 값이 같으면 메시지가 바뀌지 않았고, 비밀키를 아는 사람이 만든 메시지라고 판단할 수 있다.

즉, MAC은 **무결성 + 인증**을 제공한다. 하지만 MAC은 부인방지를 제공하지 못한다. 왜냐하면 송신자와 수신자가 같은 비밀키를 공유하기 때문에, 제3자 입장에서는 둘 중 누가 MAC을 만들었는지 구분하기 어렵기 때문이다.

# HMAC
HMAC은 Hash-based MAC으로 암호학적 해시 함수를 기반으로 MAC을 구성하는 방식이다.

HMAC을 사용하는 이유는 해시 함수가 소프트웨어에서 빠르게 동작하고, MD5나 SHA 같은 해시 함수 라이브러리가 널리 제공되기 때문이다.

개념적으로 HMAC은 “비밀키와 메시지를 함께 사용해서 해시 기반 인증값을 만드는 방식”이다.

단순히 `H(key || message)`처럼 붙이는 방식은 안전하지 않을 수 있기 때문에, 실제 HMAC은 내부 패드와 외부 패드를 사용하여 구성된다.

![](../images/Pasted%20image%2020260518234852.png)

정리하면 HMAC은 해시 함수의 빠른 계산성과 MAC의 인증 기능을 결합한 방식이다.

# Digital Signature
전자서명은 MAC과 비슷하게 메시지 인증을 제공하지만, 사용하는 키 구조가 다르다.

- MAC은 **대칭키** 기반이다.  
- 전자서명은 **비대칭키** 기반이다.

전자서명에서는 송신자가 자신의 **개인키**로 서명하고, 검증자는 송신자의 **공개키**로 검증한다.

일반적인 흐름은 다음과 같다.

1. 메시지를 해시한다.  
2. 해시값에 대해 개인키로 서명한다.  
3. 메시지와 서명을 함께 보낸다.  
4. 검증자는 메시지를 다시 해시하고, 공개키로 서명을 검증한다.

이 구조를 사용하면 메시지가 바뀌었는지 확인할 수 있고, 해당 개인키 소유자가 서명했다는 것도 확인할 수 있다. 또한 공개키로 누구나 검증할 수 있으므로, 제3자에게도 증명 가능하다. 이것이 부인방지로 이어진다.

---
## Use of Digital Signature
전자서명은 보통 메시지 전체를 직접 서명하지 않는다. 메시지를 먼저 해시한 뒤, 그 해시값에 서명한다.

이유는 두 가지이다.

- 공개키 연산은 느리다. 메시지 전체에 대해 직접 서명하면 비효율적이다.
- 해시값은 고정 길이이다. 메시지 크기와 관계없이 일정한 크기의 해시값만 서명하면 되므로 처리하기 좋다.

![](../images/Pasted%20image%2020260518235303.png)

검증자는 받은 메시지를 다시 해시하고, 서명을 공개키로 검증해서 나온 값과 비교한다. 두 값이 같으면 메시지가 서명 이후 변경되지 않았고, 서명자의 개인키로 만들어진 서명임을 확인할 수 있다.

---
## Key Process
전자서명 과정에서는 개인키와 공개키가 서로 다른 역할을 한다.

- 서명자는 개인키를 사용한다.  
- 검증자는 공개키를 사용한다.

예를 들어 Alice가 Bob에게 서명된 메시지를 보낸다고 하자. Alice는 자신의 개인키로 서명하고, Bob은 Alice의 공개키로 검증한다.

![](../images/Pasted%20image%2020260518235741.png)

여기서 중요한 문제는 Bob이 가지고 있는 공개키가 정말 Alice의 공개키인지 확인해야 한다는 점이다. 만약 공격자가 자신의 공개키를 Alice의 공개키처럼 속이면, Bob은 공격자의 서명도 Alice의 서명으로 착각할 수 있다. 그래서 전자서명에서는 단순히 공개키만 있는 것이 아니라, 공개키의 소유자를 증명하는 인증서와 PKI가 필요하다.

---
## Digital Signature Requirements
전자서명은 개인키 소유자가 문서에 서명하고, 공개키를 아는 누구나 그 서명을 검증할 수 있게 하는 구조이다.

전자서명은 다음 조건을 만족해야 한다.

- 서명자를 확인할 수 있어야 한다.  
- 서명 시점의 문서 내용을 인증할 수 있어야 한다.  
- 제3자가 검증할 수 있어야 한다.  
- 공개키와 기존 서명을 알고 있어도 새로운 서명을 위조할 수 없어야 한다.

이 요구사항 때문에 전자서명은 단순 인증을 넘어서 법적·기술적 증명 수단으로 사용될 수 있다. 특히 제3자가 검증할 수 있다는 점이 MAC과 다르다. MAC은 같은 비밀키를 공유한 사람끼리만 확인할 수 있지만, 전자서명은 공개키만 있으면 누구나 검증할 수 있다.

---
## Digital Signature Calculation Requirements

- 전자서명 알고리즘은 정상 사용자에게는 쉬워야 하고, 공격자에게는 어려워야 한다.
- 서명을 생성하는 과정은 쉬워야 한다. 개인키를 가진 사람은 효율적으로 서명할 수 있어야 한다.
- 서명을 검증하는 과정도 쉬워야 한다. 공개키를 가진 사람은 빠르게 서명을 검증할 수 있어야 한다.
- 반면 서명을 위조하는 것은 계산적으로 불가능해야 한다. 공격자가 공개키를 알고 있거나 일부 메시지와 서명 쌍을 알고 있더라도, 새로운 메시지에 대한 유효한 서명을 만들 수 없어야 한다.

즉, 전자서명은 “생성·검증은 쉽고, 위조는 어려운 구조”여야 한다.

---
## Signature Attack
전자서명에 대한 공격 모델은 공격자가 어떤 정보를 가지고 있는지에 따라 나뉜다.

- **Key-only attack**은 공격자가 공개키만 알고 있는 상황이다. 가장 기본적인 공격 모델이다.
- **Known message attack**은 공격자가 여러 메시지와 그에 대한 서명 쌍을 알고 있는 상황이다.
- **Generic chosen message attack**은 공격자가 서명을 받을 메시지 목록을 먼저 정하고, 그 메시지들에 대한 서명을 얻는 상황이다. 이때 공격자가 목표로 하는 것은 새로운 유효 서명을 만들어내는 것이다.

더 강한 chosen message attack도 있다.

- **Directed chosen message attack**은 공격자가 공개키를 본 뒤 서명을 요청할 메시지를 고르는 상황이다.
- **Adaptive chosen message attack**은 공격자가 이전에 받은 메시지-서명 쌍을 보고, 그 결과에 따라 다음 서명 요청 메시지를 선택할 수 있는 상황이다.

Adaptive chosen message attack이 가장 강한 공격 모델에 가깝다. 실제 안전한 전자서명 방식은 이런 강한 공격 상황에서도 위조가 어려워야 한다.

---
## Forgery
전자서명 위조는 위조의 수준에 따라 나뉜다.

- **Total break**는 공격자가 개인키를 알아낸 경우이다. 가장 심각한 공격이다.

- **Universal forgery**는 개인키를 직접 알아내지는 않았지만, 임의의 메시지에 대해 서명을 만들 수 있는 알고리즘을 얻은 경우이다.

- **Selective forgery**는 공격자가 특정 메시지를 선택하고, 그 메시지에 대한 유효한 서명을 만들어낸 경우이다.

- **Existential forgery**는 공격자가 어떤 메시지 하나라도 유효한 메시지-서명 쌍을 만들어낸 경우이다. 이때 메시지는 공격자가 원하는 의미 있는 메시지가 아닐 수도 있다.

전자서명에서는 existential forgery도 막아야 한다. 왜냐하면 유효한 서명 쌍을 하나라도 임의로 만들 수 있다는 것은 서명 구조 자체에 취약점이 있다는 뜻이기 때문이다.

---
## RSA Signature
RSA 서명은 RSA의 수학적 구조를 이용한다.

서명은 개인키 `d`로 수행한다.

> Sig<sub>sk</sub>(m) → s ≡ m<sup>d</sup> ( mod n )

검증은 공개키 `e`로 수행한다.

> Ver<sub>pk</sub>(s) → m' ≡ s<sup>e</sup> mod n

검증자는 `m = m'`인지 확인한다.

- RSA 암호화에서는 공개키로 암호화하고 개인키로 복호화한다.  
- RSA 서명에서는 개인키로 서명하고 공개키로 검증한다.

다만 이 형태는 “교과서적 RSA 서명”이고, 실제로는 메시지 자체가 아니라 해시값과 패딩을 사용해야 안전하다.

---
## Plain RSA Signature의 문제

단순 RSA 서명은 위조 공격에 취약하다. 공격자는 공개키 `(N, e)`만 알고 있어도 임의의 서명값 `s`를 고를 수 있다. 그리고 다음을 계산한다.

> m ≡ s<sup>e</sup> mod n

그러면 `(m, s)`는 유효한 메시지-서명 쌍이 된다. 왜냐하면 검증할 때 s<sup>e</sup> mod n 이 그대로 `m`이 되기 때문이다. 이것이 key-only attack에서의 existential forgery이다.

또한 RSA에는 곱셈적 성질이 있다.

![](../images/Pasted%20image%2020260519001527.png)

서명 s<sub>1</sub> ≡ m<sub>1</sub><sup>d</sup> mod n, s<sub>2</sub> ≡ m<sub>2</sub><sup>d</sup> mod n 이 있으면, s<sub>1</sub> · s<sub>2</sub> ≡ (m<sub>1</sub> · m<sub>2</sub>)<sup>d</sup> mod n 이 된다.

즉, 두 개의 서명을 곱해서 새로운 메시지에 대한 서명을 만들 수 있다. 따라서 메시지 자체를 바로 RSA로 서명하는 것은 안전하지 않다.

---
## RSA Signature with Hash Function

RSA 서명에서는 메시지 자체가 아니라 메시지의 해시값에 서명한다.

서명은 다음과 같다.

> s ≡ h(m)<sup>d</sup> mod n

검증은 다음과 같다.

> h(m)' ≡ s<sup>e</sup> mod n

검증자는 직접 계산한 `h(m)`과 서명 검증 결과로 나온 `h(m)'`를 비교한다. 이렇게 하면 메시지가 길어도 고정 길이 해시값만 서명하면 되므로 효율적이다. 또한 단순 RSA 서명의 곱셈적 위조 문제를 줄일 수 있다. 

왜냐하면 일반적으로 h(m<sub>1</sub>) · h(m<sub>2</sub>) = h(m<sub>1</sub> · m<sub>2</sub>) 가 성립하지 않기 때문이다.
자료에서도 RSA 서명을 해시 함수와 함께 사용하는 방식과 검증 과정을 따로 정리한다.

---

## Hash를 붙인 RSA Signature의 효과

해시를 사용하면 단순 RSA 서명의 위조가 어려워진다. 공격자가 임의의 서명값 `s`를 고르고 m = s<sup>e</sup> mod n을 계산할 수는 있다. 하지만 이제 검증 조건은 `m = m'`가 아니다.

검증 조건은 다음과 같다.

`h(m) = m'`

즉, 공격자는 `m'`이라는 값이 어떤 메시지의 해시값이 되도록 메시지 `m`을 찾아야 한다. 이것은 preimage resistance 때문에 어렵다.

또한 기존 서명 두 개를 곱하더라도 결과는 (h(m1) · h(m2))<sup>d</sup> mod n 형태가 된다.

이 값이 어떤 메시지 m<sub>3</sub>에 대한 h(m<sub>3</sub>)<sup>d</sup> mod n이 되려면 h(m<sub>3</sub>) = h(m1) · h(m2)가 되어야 하는데, 이런 메시지를 찾는 것은 어렵다.

따라서 RSA 서명에서는 해시 함수가 위조 방지에 중요한 역할을 한다. 자료에서도 해시를 붙인 RSA 서명은 preimage resistance와 관련해 위조가 어려워진다고 설명한다.

# ElGamal Signature

ElGamal 서명은 이산 로그 문제에 기반한 전자서명 방식이다.

키는 다음과 같이 구성된다.

![](../images/Pasted%20image%2020260519002455.png)

여기서 `x`는 개인키이고, `y, g, p`는 공개키이다.

서명할 때는 랜덤한 정수 `k`를 선택한다. 이 `k`는 매번 새롭게 선택되어야 하고, 절대 재사용되면 안 된다.

서명값은 두 부분으로 구성된다.

![](../images/Pasted%20image%2020260519002515.png)

따라서 서명은 `(γ, δ)`이다.

검증은 다음 식을 확인하는 방식이다.

![](../images/Pasted%20image%2020260519002541.png)

이 식이 성립하는 이유는 `y = g^x`, `γ = g^k`이기 때문이다. 오른쪽을 정리하면 `g^(xγ + kδ)`가 되고, δ의 정의에 의해 지수가 `m`과 같아진다.

---

# 슬라이드 37. ElGamal Signature Example

예제에서는 다음 값을 사용한다.

`p = 113`, `g = 3`, `x = 42`

공개키는 다음과 같이 계산된다.

`y = g^x = 3^42 mod 113 = 69`

따라서 공개키는 `(69, 3, 113)`이고, 개인키는 `42`이다.

메시지는 `m = 26`, 랜덤값은 `k = 23`이다.

먼저

`γ = 3^23 mod 113 = 39`

그리고

`δ = (m - xγ) k^-1 mod 112`

계산 결과 `δ = 76`이다.

따라서 서명은 `(39, 76)`이다.

검증에서는 다음을 비교한다.

`g^m = 3^26 mod 113 = 36`

그리고

`y^γ γ^δ = 69^39 · 39^76 mod 113 = 36`

두 값이 같으므로 서명이 유효하다.

---

# 슬라이드 38. ElGamal Signature의 위조

이 슬라이드는 ElGamal 서명에서도 단순한 형태를 그대로 사용하면 existential forgery가 가능할 수 있음을 보여준다.

공격자가 공개키만 알고 있어도 특정한 방식으로 임의의 값 `i`, `j`를 선택해서 어떤 메시지와 그에 대한 서명 쌍을 만들어낼 수 있다는 내용이다.

핵심은 단순한 수학적 서명 구조만으로는 충분하지 않다는 점이다.

그래서 실제 전자서명에서는 메시지 자체를 직접 서명하지 않고 해시값을 사용하며, 표준화된 안전한 서명 알고리즘을 사용한다.

즉, ElGamal 서명도 개념적으로는 중요하지만, 실제 사용에서는 해시 함수, 랜덤값 관리, 검증 조건 등이 함께 고려되어야 한다.

# 슬라이드 39. Schnorr Signature

Schnorr 서명은 이산 로그 문제 기반의 전자서명 방식이다.

파라미터는 다음과 같다.

`p`: 큰 소수  
`q`: `p - 1`을 나누는 소수  
`g`: order가 `q`인 원소  
`x`: 개인키  
`y = g^x mod p`: 공개키

서명할 때는 랜덤값 `k`를 선택한다.

서명은 다음과 같이 구성된다.

`γ = h(m || g^k mod p)`

`δ = k + xγ mod q`

서명은 `(γ, δ)`이다.

검증자는 다음 값을 계산한다.

`γ' = h(m || g^δ y^-γ mod p)`

그리고 `γ = γ'`인지 확인한다.

이게 가능한 이유는 다음과 같다.

`g^δ y^-γ = g^(k + xγ) · (g^x)^-γ`

이를 정리하면

`g^(k + xγ - xγ) = g^k`

가 된다.

즉, 검증자는 랜덤값 `k`를 직접 모르더라도 `g^k`에 해당하는 값을 복원해서 해시 비교를 할 수 있다.

---

# 슬라이드 40. DSA

DSA는 Digital Signature Algorithm이다.

DSA도 이산 로그 문제 기반의 서명 방식이다.

키는 다음과 같이 구성된다.

`p`, `q`, `g`는 공개 파라미터이다.  
`x`는 개인키이다.  
`y = g^x mod p`는 공개키이다.

서명할 때는 랜덤값 `k`를 선택한다.

서명값은 `(γ, δ)`이다.

`γ = (g^k mod p) mod q`

`δ = (h(m) + xγ) k^-1 mod q`

검증할 때는 다음 값을 계산한다.

`e1 = h(m) δ^-1 mod q`

`e2 = γ δ^-1 mod q`

그 후

`γ' = (g^e1 y^e2 mod p) mod q`

를 계산해서 `γ = γ'`인지 확인한다.

DSA에서 특히 중요한 것은 랜덤값 `k`이다. `k`가 재사용되거나 예측 가능하면 개인키 `x`가 노출될 수 있다.

---

# 슬라이드 41. ECDSA

ECDSA는 Elliptic Curve Digital Signature Algorithm이다.

DSA를 타원곡선 그룹 위에서 구현한 방식으로 이해할 수 있다.

키는 다음과 같이 구성된다.

`G`는 타원곡선의 기준점이다.  
`n`은 `G`의 order로, `nG = O`를 만족한다.  
`dA`는 개인키이다.  
`QA = dA × G`는 공개키이다.

서명할 때는 랜덤값 `k`를 선택하고,

`k × G = (x1, y1)`

을 계산한다.

그다음 서명값을 만든다.

`γ = x1 mod n`

`δ = (h(m) + dA · γ) k^-1 mod n`

서명은 `(γ, δ)`이다.

검증자는 다음 값을 계산한다.

`e1 = h(m) δ^-1 mod n`

`e2 = γ δ^-1 mod n`

그다음

`e1G + e2QA = (x1', y1')`

를 계산하고,

`γ ≡ x1' mod n`

이면 서명이 유효하다고 판단한다.

ECDSA는 RSA보다 짧은 키로 비슷한 보안 강도를 제공할 수 있어서 현대 시스템에서 많이 사용된다. 자료에서도 ECDSA의 키, 서명, 검증 과정을 타원곡선 기준점 `G`, 개인키 `dA`, 공개키 `QA`를 중심으로 정리한다.

---

# 슬라이드 42. PKI

PKI는 Public Key Infrastructure, 즉 공개키 기반 구조이다.

전자서명에서는 공개키로 서명을 검증한다. 그런데 여기서 중요한 문제가 있다.

검증자가 가진 공개키가 정말 서명자의 공개키인지 어떻게 확인할 수 있는가?

이 문제를 해결하는 것이 인증서와 CA이다.

CA는 Certificate Authority, 즉 인증기관이다. CA는 어떤 주체의 공개키와 신원 정보를 묶어서 인증서를 만들고, 그 인증서에 자신의 개인키로 서명한다.

인증서에는 다음과 같은 정보가 포함된다.

Version  
Serial Number  
Signature Algorithm Identifier  
Issuer Name  
Validity Period  
Subject Name  
Public Key Information  
Extensions  
Signature

즉, 인증서는 “이 공개키는 이 주체의 것이다”라는 사실을 CA가 보증한 문서이다. 자료에서도 PKI를 CA, 인증서, 공개키, X.509 v3 Certificate 구조로 설명한다.

---

# 슬라이드 43. Certificate Chain - Digital Signature

인증서도 결국 전자서명을 통해 검증된다.

CA는 인증서 내용을 해시하고, 그 해시값에 대해 자신의 개인키로 서명한다.

검증자는 CA의 공개키를 사용해서 인증서의 서명을 검증한다.

검증 과정은 다음과 같다.

먼저 인증서 본문을 해시한다.  
그다음 인증서에 포함된 서명을 CA 공개키로 검증한다.  
검증 결과로 나온 해시값과 직접 계산한 해시값을 비교한다.

두 값이 같으면 인증서 내용이 변조되지 않았고, 해당 CA가 서명한 인증서임을 확인할 수 있다.

즉, 인증서의 신뢰성은 전자서명에 기반한다.

---

# 슬라이드 44. Certificate Chain

실제 인증서는 하나의 CA만으로 끝나지 않고 체인 구조를 가진다.

가장 위에는 **Root CA**가 있다. Root CA는 운영체제나 브라우저에 미리 신뢰할 수 있는 기관으로 등록되어 있다. 이를 trust anchor라고 한다.

Root CA 아래에는 **Intermediate CA**가 있을 수 있다.  
Intermediate CA는 Root CA 또는 상위 CA로부터 인증서를 발급받는다.

마지막에는 실제 서버나 사용자에게 발급된 **Leaf Certificate**가 있다.

검증자는 Leaf Certificate부터 시작해서 상위 인증서의 서명을 차례대로 검증한다. 최종적으로 신뢰 가능한 Root CA에 도달하면 인증서 체인이 유효하다고 판단한다.

이 구조는 HTTPS/TLS에서 서버 인증에 사용된다.

# 슬라이드 45. Hash vs MAC vs Digital Signature

이 표는 9장의 핵심 정리이다.

|구분|Hash|MAC|Digital Signature|
|---|---|---|---|
|Data Integrity|가능|가능|가능|
|Authentication|불가능|가능|가능|
|Non-repudiation|불가능|불가능|가능|
|Key type|없음|대칭키|비대칭키|

해시는 데이터가 바뀌었는지만 확인할 수 있다. 누구나 해시값을 만들 수 있으므로 인증은 제공하지 못한다.

MAC은 비밀키를 사용하는 방식이다. 비밀키를 아는 사람만 올바른 MAC을 만들 수 있으므로 인증이 가능하다. 하지만 송신자와 수신자가 같은 키를 공유하므로 제3자에게 누가 만들었는지 증명하기 어렵다. 따라서 부인방지는 제공하지 못한다.

전자서명은 개인키로 서명하고 공개키로 검증한다. 개인키는 서명자만 가지고 있고, 공개키로 누구나 검증할 수 있으므로 무결성, 인증, 부인방지를 모두 제공한다.

이 표는 시험에 그대로 나올 수 있을 정도로 중요하다. 자료에서도 Hash, MAC, Digital Signature를 data integrity, authentication, non-repudiation, key type 기준으로 비교한다.
